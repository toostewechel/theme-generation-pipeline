# Figma Token Export — Design Spec

*Written 2026-06-22. A studio feature that copies the generated token set to the clipboard in the project's native DTCG + manifest format, so it can be imported into Figma (by a future custom plugin, or by the Figma MCP today) and round-trips with the existing pipeline.*

## Goal

Add a **"Copy tokens for Figma"** button to the color studio. It serializes the current theme's generated tokens into a single JSON blob that mirrors the project's on-disk token layout (`manifest.json` + the DTCG token files). That blob is the input a Figma importer consumes.

The export is **lossless and identical to what `build:theme` writes**: colors stay in `oklch` colorSpace, semantic tokens stay as `{reference}` aliases. Conversion to a Figma-ingestible color representation (sRGB hex) is the *importer's* responsibility, not the exporter's.

## Non-goals

- No Figma import code ships in this feature. Importing is done by the Figma MCP (today, via an agent workflow) or a future custom plugin. This spec documents the ingestion mapping but builds no importer.
- No Tokens Studio (or other vendor) dialect. The export is the project's own DTCG format.
- No hex/sRGB conversion in the exporter. oklch values are emitted verbatim.
- No non-color collections (radius, dimension, typography, effects). Color only — the part the studio generates. The bundle shape is designed to extend to these later.
- **Amended during implementation:** the static prism palette (`primitives-color.static.tokens.json`) IS carried in the bundle as a verbatim passthrough. It was originally out of scope, but the semantic `color-retro-*` and `color-circled-marker` tokens alias `{color-prism-*}`, so the bundle must include prism for every alias to resolve (the self-contained-bundle constraint). It is carried as-is, not generated; pipeline output is unchanged.
- No file download. Clipboard only.

## Background

The studio (`tools/color-studio`) generates the full color system in-browser from `ThemeInputs` using the shared, isomorphic engine in `src/engine`. The CLI path (`scripts/buildTheme.ts` → `src/engine/emit-dtcg.ts`) already emits three DTCG files consumed by the Style Dictionary build:

- `primitives-color.mode-1.tokens.json` — primitive ramps, alphas, dark surfaces, brand. Color values are objects: `{ "colorSpace": "oklch", "components": [l, c, h] }` (with optional `alpha`).
- `color.light.tokens.json` / `color.dark.tokens.json` — semantic tokens as string references: `"{color-neutral-700}"`.

`src/tokens/manifest.json` indexes these as `collections → modes → [files]`, a structure that maps 1:1 onto Figma variable collections and modes. That manifest is the canonical description of the system's collection/mode layout.

This feature reuses that emitter and that manifest structure rather than inventing a new format, so the export is a single source of truth shared by the pipeline, a future import plugin, and the MCP.

## Architecture

### Refactor: separate token assembly from disk writing (`src/engine/emit-dtcg.ts`)

`emit-dtcg.ts` currently assembles token objects (`buildPrimitivesDtcg`, `buildSemanticDtcg`) and `writeGeneratedTokens` writes them to disk with their canonical filenames. Extract the in-memory assembly so both the disk writer and the studio bundle share one path:

```ts
// returns the generated token files keyed by their canonical filename,
// each including the same "$description" header the pipeline writes today
export function buildGeneratedFiles(inputs: ThemeInputs): Record<string, object> {
  const desc = "auto-generated by build:theme — do not edit";
  return {
    "primitives-color.mode-1.tokens.json": { $description: desc, ...buildPrimitivesDtcg(inputs) },
    "color.light.tokens.json": { $description: desc, ...buildSemanticDtcg(inputs, "light") },
    "color.dark.tokens.json": { $description: desc, ...buildSemanticDtcg(inputs, "dark") },
  };
}
```

`buildGeneratedFiles` owns the `$description` header (moved out of `writeGeneratedTokens`), and `writeGeneratedTokens` is reimplemented to iterate it and write each entry verbatim. This guarantees the bytes the studio copies equal the bytes the pipeline writes — one assembly path, no drift. (Confirm during implementation exactly where the header is added today and move it here without changing the emitted output.)

### New: `src/engine/figma-export.ts` (pure / isomorphic)

Culori-only, no Node imports, so it passes the existing isomorphism guard test and runs in both the browser and the CLI.

```ts
export interface TokenBundle {
  manifest: { name: string; collections: Record<string, { modes: Record<string, string[]> }> };
  files: Record<string, object>;
}

export function buildTokenBundle(inputs: ThemeInputs): TokenBundle;
export function serializeTokenBundle(inputs: ThemeInputs): string; // JSON.stringify(bundle, null, 2)
```

`buildTokenBundle`:
1. `files = buildGeneratedFiles(inputs)`.
2. `manifest` = the color portion of the canonical manifest, as an embedded constant `COLOR_MANIFEST` (see Format below). A test asserts `COLOR_MANIFEST` matches the corresponding collections in `src/tokens/manifest.json` (drift guard), keeping the module pure while staying in sync.

### Studio wiring

- `tools/color-studio/index.html` — add a button `#export-figma-btn` in the header, next to `#save-btn`.
- `tools/color-studio/src/export-figma.ts` (new, small) — `copyTokensForFigma(state: ThemeInputs): Promise<boolean>` calls `serializeTokenBundle(state)` and `navigator.clipboard.writeText(...)`, returning success.
- `tools/color-studio/src/main.ts` — wire the button: on click, call the helper and show transient feedback (`Copied ✓` / `Copy failed`, reverting after ~1.6s), mirroring the existing save-button pattern. The export reflects the live `state`, independent of whether it has been saved to config.

## Format

A single clipboard JSON mirroring the on-disk layout:

```jsonc
{
  "manifest": {
    "name": "Design Tokens",
    "collections": {
      "primitives-color": { "modes": { "mode-1": ["primitives-color.mode-1.tokens.json"] } },
      "color":            { "modes": { "light": ["color.light.tokens.json"],
                                       "dark":  ["color.dark.tokens.json"] } }
    }
  },
  "files": {
    "primitives-color.mode-1.tokens.json": {
      "color-neutral-700": { "$type": "color",
        "$value": { "colorSpace": "oklch", "components": [0.515, 0.006, 75] } }
      // … all primitive ramps, black/white alpha (with "alpha"), dark surfaces, brand
    },
    "color.light.tokens.json": {
      "color-bg-accent": { "$type": "color", "$value": "{color-accent-500}" }
      // … all semantic color tokens, light mode
    },
    "color.dark.tokens.json": {
      "color-bg-accent": { "$type": "color", "$value": "{color-accent-500}" }
      // … all semantic color tokens, dark mode
    }
  }
}
```

Notes:
- `manifest` is `COLOR_MANIFEST`: only the `primitives-color` and `color` collections (the generated ones). `name` matches the canonical manifest (`"Design Tokens"`).
- `files` keys are the canonical filenames the manifest references, so an importer resolves `manifest → files` exactly as the build resolves `manifest → disk`.
- Color values are oklch objects, verbatim from the emitter. Semantic values are `{reference}` strings. No hex anywhere in the export.
- Non-color semantic tokens that are not `$type: color` (e.g. state-intensity number tokens emitted as literal DTCG objects) flow through `buildSemanticDtcg` exactly as they do to disk today — they are not stripped, because the export equals the pipeline files. The importer decides how to handle non-color tokens (it may create number variables or skip them).

## Importer ingestion (documented, not shipped)

The Figma MCP today, or a future custom plugin, consumes the bundle as follows:

1. For each `collection` in `manifest.collections`, create/locate a Figma variable collection of that name.
2. For each `mode` under the collection, create/locate that Figma mode.
3. Concatenate the token objects from the mode's listed `files`. For each token:
   - **Object color value** (`colorSpace: "oklch"`): convert to sRGB hex with culori — `formatHex` for opaque, `formatHex8` when `alpha` is present. The colors are already P3-gamut-clamped by the engine; mapping to sRGB hex may clip the most saturated P3 colors. This clipping is **by design** and happens only at import — the source export stays lossless.
   - **Reference value** (`"{token-name}"`): set the Figma variable as an alias to the variable of that name (resolved within the imported collections).
4. Cross-collection aliases (semantic `color` variables referencing `primitives-color` variables) are wired after both collections exist.

## Testing

`src/engine/figma-export.test.ts`:

1. **Bundle shape:** result has `manifest` and `files`; `files` contains exactly the three canonical filenames.
2. **Emitter parity (drift guard):** `bundle.files` deep-equals `buildGeneratedFiles(inputs)` for a representative `ThemeInputs`.
3. **Manifest parity (drift guard):** `bundle.manifest.collections` for `primitives-color` and `color` deep-equals those collections in `src/tokens/manifest.json` (read in the test's Node context).
4. **Alias resolution:** every `{ref}` value across `color.light`/`color.dark` names a token present in `primitives-color.mode-1` (or another generated file).
5. **Mode divergence:** at least one semantic token differs between `color.light` and `color.dark`.
6. **Value shape:** primitive color values are oklch objects (or srgb for any static passthrough if added later); no value in any generated file is a hex string (guards against accidental conversion in the exporter).

The existing isomorphism guard test must continue to pass with `figma-export.ts` added (no `fs`/`path`/`node:` imports).

## Risks / trade-offs

- **Not directly Figma-ingestible.** The export needs an importer to convert oklch→sRGB. Accepted: the future plugin / MCP owns that conversion; the payoff is one lossless format shared with the pipeline.
- **P3→sRGB clipping at import.** The studio preview (P3 `oklch()`) can read marginally more vivid than the Figma result. Accepted; a future plugin could carry P3.
- **Clipboard is one blob, the pipeline is many files.** The bundle wraps files under a `files` map; importers unbundle via the manifest. Minor structural wrinkle, not a fidelity loss.
- **Embedded `COLOR_MANIFEST` could drift from `manifest.json`.** Mitigated by the manifest-parity test.

## Future extensions (out of scope)

- Add the static prism palette (`primitives-color.static.tokens.json`) and non-color collections (radius/dimension/typography) to the bundle by extending `COLOR_MANIFEST` and `buildGeneratedFiles` — the shape already supports it.
- A first-party Figma import plugin that reads this bundle directly.
- Round-trip import *back into the pipeline* (the bundle is already the pipeline's input format).
